Synchro currently ships three client surfaces:

- Swift for Apple platforms
- Kotlin for Android
- React Native as a bridge over the native SDKs

## Native SDK responsibilities

The native SDKs own:

- local SQLite access
- local mutation capture
- durable queueing
- background and foreground sync scheduling
- applying server state without echo loops

They do not define server sync semantics.

## React Native boundary

React Native exposes the same product surface as the native SDKs, but it must remain a bridge.

It does not own:

- queue semantics
- rebuild semantics
- conflict semantics
- server protocol interpretation

## Installation notes

- Swift is released through Swift Package Manager from the repo root `Package.swift`
- Kotlin is released from `clients/kotlin/synchro`
- React Native is released from `clients/react-native`

## Seeded database support

The native SDKs support copying a preinitialized SQLite database into place before first open.

Use this for:

- warm-starting the synced-table schema
- bundling reference data or other installer-time content you want available immediately
- reducing first-start schema work and initial traffic

Generate the seed artifact with:

- `api/go/cmd/synchro-seed`

The generated database is still only a warm cache. The SDKs always reconcile it against the authoritative server contract on first `connect`.

### React Native on iOS

The current iOS integration requires the consuming app to add the native Synchro dependency and GRDB in its Podfile. This is part of the current public installation contract until the CocoaPods dependency story is simplified further.

```ruby
pod 'Synchro', :git => 'https://github.com/trainstar/synchro.git', :tag => 'v0.2.0'
pod 'GRDB.swift', :git => 'https://github.com/groue/GRDB.swift.git', :tag => 'v7.0.0'
```
