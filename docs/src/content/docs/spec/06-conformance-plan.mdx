# Conformance Plan

## Purpose

This document defines the executable validation strategy for Synchro.

The goal is to turn the spec into a real contract that every layer must prove, not a prose document that each implementation interprets differently.

## Core Rule

The source of truth is:

- `00-principles`
- `01-wire-protocol`
- `02-client-contract`
- `03-state-machines`
- `04-invariants`
- `05-schema-evolution`

Current implementation behavior is not baseline.

If code disagrees with the spec, the code is wrong unless the spec is deliberately changed.

## What Conformance Must Prove

Conformance must prove:

- wire compatibility
- deterministic sync semantics
- correct state-machine behavior
- correct rebuild behavior
- correct schema evolution behavior
- correct local metadata and queue durability
- parity across adapters and end clients
- performance characteristics that justify the architecture

## Test Asset Model

The conformance suite should use shared executable assets, not only handwritten tests per layer.

Recommended shared asset groups:

- protocol fixtures
- schema fixtures
- scope and rebuild fixtures
- mutation and conflict fixtures
- state-machine traces
- performance scenarios

Recommended repository shape:

```text
conformance/
  protocol/
  schema/
  scopes/
  mutations/
  traces/
  performance/
```

These assets should be reusable by:

- `extensions/synchro-core`
- `extensions/synchro-pg`
- `api/go`
- Swift client tests
- Kotlin client tests
- React Native bridge tests where relevant

## Conformance Layers

### Layer 1: Shared Fixture Validation

These are golden inputs and expected outputs that define contract truth.

Examples:

- `connect` request and response pairs
- `push` acceptance and rejection cases
- `pull` delta and rebuild requests
- schema action cases
- state transition traces

### Layer 2: Rust Core Deterministic Semantics

`extensions/synchro-core` should prove:

- cursor semantics
- version-token handling rules
- checksum behavior
- diff and dedup behavior
- mutation outcome normalization
- state-machine transition logic that belongs in shared semantics

These tests should be mostly pure and fixture-driven.

### Layer 3: PostgreSQL Extension Integration

`extensions/synchro-pg` should prove:

- WAL or CDC-driven change capture
- deterministic scope materialization
- low-fanout shared-scope behavior
- per-scope incremental serving
- rebuild correctness
- synced CRUD application
- schema manifest generation
- schema action selection

These tests must run against real PostgreSQL.

### Layer 4: HTTP Adapter Conformance

`api/go` should prove:

- exact endpoint behavior from `01-wire-protocol`
- auth and version-gating integration
- error translation
- idempotency behavior
- compression and transport behavior where supported

These tests should treat the adapter as a black box over the extension.

The adapter should not need its own semantic test universe.

### Layer 5: Native Client Conformance

Swift and Kotlin should prove:

- local SQLite materialization
- queue durability
- safe apply without echo-loop
- push before pull ordering
- rebuild correctness
- schema migration behavior
- seeded DB warm-start behavior
- offline writes before first connect
- typed status and error surface

These tests should use the same shared contract scenarios wherever possible.

### Layer 6: React Native Bridge Conformance

React Native should prove:

- parity of the public SDK surface
- typed event and error forwarding
- no independent sync semantics

React Native does not need a separate deep sync logic suite if native clients already prove the semantics.

## Scenario Matrix

Every major spec area should map to automated scenarios.

### Wire Protocol Scenarios

- successful `connect` with no schema change
- `connect` with `replace`
- `connect` with `rebuild_local`
- `connect` with `unsupported`
- successful `push`
- mixed accepted and rejected `push`
- `pull` with delta only
- `pull` with scope add and remove
- `pull` requesting rebuild
- paged `rebuild`

### State-Machine Scenarios

- empty first startup
- seeded startup
- offline write before first connect
- push then pull steady state
- retryable failure during connect
- retryable failure during pull
- non-retryable incompatibility
- stop during rebuild

### Invariant Scenarios

- no silent mutation loss
- no server-apply echo loop
- cursor advances only after durable apply
- rebuild does not erase overlapping valid rows
- rejected mutation remains inspectable
- scope assignment remains server-authoritative

### Schema Evolution Scenarios

- additive column
- additive table with historical rows
- single-scope to multi-scope transition
- scope-logic change without structural schema change
- phased rollout preserving queued mutations
- unsupported destructive change

### Shared Versus Private Scope Scenarios

- one private scope per row
- shared public scope with many subscribers
- row present in both shared and private scopes
- public-row update with low server-side fanout

## Performance Conformance

Correctness is primary, but performance claims must also be tested.

### Round-Trip Budget

The suite should assert the intended hot-path request budget:

- warm connect: 1 request
- steady-state sync with no pending local mutations: 1 pull request per cycle
- steady-state sync with pending local mutations: 1 push request plus 1 pull request per cycle
- rebuild: explicit extra requests only for rebuild pagination

### Server-Side Workload Expectations

The suite should measure and report:

- per-row fanout distribution
- shared-scope versus duplicated-private-scope behavior
- rebuild cost by scope size
- schema-check handshake cost

### Client-Side Expectations

The suite should measure and report:

- seeded startup versus empty startup
- queue replay cost
- rebuild apply cost
- local provenance overhead for multi-scope tables

## Required Test Style

### Shared Fixtures First

Every important semantic rule should exist first as a fixture or scenario definition, then as layer-specific tests consuming it.

### End-To-End For Silent Failure Zones

For silent failure zones, tests should prefer full-flow validation over brittle unit isolation.

Key silent-failure zones include:

- synced CRUD reconciliation
- rebuild
- schema migration
- provenance pruning
- scope cursor durability
- offline queue replay

### Black-Box Adapter Validation

`api/go` should be validated through HTTP behavior, not by poking internal helpers.

### Real SQLite And Real PostgreSQL

Native client tests should use real SQLite.

Extension tests should use real PostgreSQL.

Do not substitute fake databases for the contract path.

## Exit Criteria

Conformance is not complete until all of the following are true:

- every invariant in `04-invariants` maps to at least one automated test
- every endpoint in `01-wire-protocol` has success, failure, and incompatibility coverage
- every schema change class in `05-schema-evolution` has automated coverage
- Swift and Kotlin pass the same core behavior corpus
- React Native proves bridge parity where applicable
- zero skipped contract tests
- zero failing contract tests
