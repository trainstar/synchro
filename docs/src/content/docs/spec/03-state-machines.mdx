# State Machines

## Purpose

This document defines the canonical client and server state machines for Synchro.

It covers:

- first startup
- steady-state sync
- rebuild
- retry and backoff
- schema replacement
- offline writes before first connect

## Core Principles

- first sync uses the same rebuild model as later recovery
- seeded or preinitialized local databases are warm starts, not a separate sync mode
- one local table may be composed from multiple scopes
- scoped rebuild uses hybrid local provenance metadata
- rebuild is automatic by default
- offline writes before first connect are allowed

## Client State Machine

### States

#### `uninitialized`

Client object exists but local database and sync metadata are not ready.

#### `local_ready`

Local database is open and client metadata tables are available.

This state may begin from:

- an empty local database
- a preinitialized or seeded local database containing schema only
- a preinitialized or seeded local database containing schema plus portable scope data

#### `connecting`

Client is negotiating protocol, schema, and scope assignment via `POST /sync/connect`.

#### `schema_applying`

Client is applying a schema replacement or schema-triggered local rebuild action.

#### `ready`

Client has an authoritative scope set and is able to run the normal sync loop.

#### `pushing`

Client is uploading queued local mutations.

#### `pulling`

Client is fetching incremental changes for assigned scopes.

#### `rebuilding`

Client is rebuilding one or more scopes explicitly requested by the server or required for first sync.

#### `backoff`

Client encountered a retryable failure and is waiting before retrying.

#### `error`

Client encountered a non-retryable failure or a failure requiring explicit application intervention.

#### `stopped`

Client sync is intentionally stopped.

## Client Startup Flow

`startSync()` should arm the client-owned startup state machine.

It must not require the host application to supervise transient startup retry with its own timers, connectivity listeners, or repeated `startSync()` calls.

### Path A: Empty Local Database

1. enter `uninitialized`
2. initialize local DB and metadata tables
3. enter `local_ready`
4. call `POST /sync/connect`
5. enter `connecting`
6. apply schema if needed
7. enter `schema_applying` if required
8. enter `ready`
9. any assigned scope without a usable cursor is marked for rebuild
10. enter `rebuilding`
11. after rebuild completion, enter `ready`
12. begin normal push and pull loop

### Path B: Seeded Or Preinitialized Local Database

1. enter `uninitialized`
2. open preinitialized local DB and metadata
3. enter `local_ready`
4. call `POST /sync/connect`
5. enter `connecting`
6. apply schema action if required
7. preserve any portable seeded scopes already present locally
8. compare local scope metadata against authoritative server scope assignment
9. rebuild only scopes that require it
10. pull incremental changes for already-seeded portable scopes as normal delta
11. resume normal push and pull loop

Seeded databases are a warm-start optimization only.

They do not create a different sync protocol mode.

Portable seeded data must already belong to server-declared portable scopes.

Portable seed generation does not use client JWT input.

## Offline Writes Before First Connect

Clients may accept offline writes to synced tables before first successful connect.

Flow:

1. local SQL write occurs
2. local mutation is captured into the durable mutation queue
3. client eventually reaches `ready`
4. client pushes queued mutations before incremental pull

This is valid local-first behavior and must be supported.

## Normal Sync Cycle

From `ready`:

1. if pending mutations exist, enter `pushing`
2. call `POST /sync/push`
3. reconcile accepted and rejected mutations
4. return to `ready`
5. enter `pulling`
6. call `POST /sync/pull`
7. apply incremental changes
8. advance per-scope cursors atomically with successful apply
9. if `has_more = true`, remain in `pulling`
10. if rebuild list is non-empty, transition to `rebuilding`
11. otherwise return to `ready`

Push-before-pull is canonical.

## Rebuild Flow

### When Rebuild Happens

Rebuild occurs when:

- a scope has no usable cursor on first sync
- the server explicitly requests rebuild for one or more scopes
- schema action requires local rebuild
- the client detects local scope state is unusable

### Rebuild Semantics

Rebuild is scope-local.

It does not imply wiping the full local database.

### Rebuild Steps

For each scope:

1. enter `rebuilding`
2. mark the scope as rebuild-in-progress in local metadata
3. begin `POST /sync/rebuild` pagination for that scope
4. write rebuilt rows into local app tables
5. write or update local scope provenance metadata
6. when rebuild completes, prune stale provenance for that scope
7. remove local app rows only when they have zero remaining provenance
8. verify the final authoritative checksum against local scope state
9. install final authoritative cursor and checksum for that scope only if verification succeeds
10. clear rebuild-in-progress marker
11. continue with next requested scope or return to `ready`

Rebuild cleanup is provenance cleanup. It must not invent a new local tombstone. If a row is absent from the rebuilt scope set and no other local provenance edge remains, the client removes the local row.

### Hybrid Provenance Model

The client uses:

- no per-row provenance metadata for tables that are guaranteed single-scope
- per-row scope provenance metadata for tables that may be composed from multiple scopes

This allows:

- cheap path for single-scope tables
- correct scoped rebuild for multi-scope tables

## Shared And Private Scope Composition

One local table may be composed from multiple scopes.

Example:

- `templates_public`
- `templates_user:u_123`

Both may materialize rows into the same local `templates` table.

The app queries the local table normally.

Scope composition is a sync-engine concern, not an application-query concern.

## Schema Flow

### Connect-Time Schema Handling

On `connect`, the server may return:

- `none`
- `fetch`
- `replace`
- `rebuild_local`
- `unsupported`

### Client Actions

#### `none`

Proceed to normal sync.

#### `fetch`

Fetch canonical schema from `GET /sync/schema`, apply it locally, then continue.

#### `replace`

Apply returned schema definition and update local metadata.

#### `rebuild_local`

Apply returned schema definition, then rebuild affected scopes before returning to normal sync.

#### `unsupported`

Enter `error` and surface upgrade-required state to the application.

## Mutation Reconciliation Flow

### Accepted Mutation

1. remove mutation from queue
2. apply canonical server row if returned
3. update local row server version metadata

### Rejected Mutation: Conflict

1. keep a rejection record
2. remove or mark the queued mutation as resolved according to the server outcome
3. apply authoritative server row if returned
4. emit conflict or rejection event

### Rejected Mutation: Validation Or Policy

1. keep a rejection record for inspection
2. do not retry automatically unless server explicitly marks retryable
3. surface typed application-visible event or inspection result

## Retry And Backoff Flow

Retryable failures move the client into `backoff`.

Examples:

- `429`
- `503`
- transport timeout
- temporary connectivity loss

Flow:

1. failure occurs in `connecting`, `pushing`, `pulling`, or `rebuilding`
2. client computes retry delay
3. enter `backoff`
4. wait
5. return to the interrupted operation or restart from `connecting` if safer

Non-retryable failures move the client into `error`.

## Stop Flow

From any active state:

1. application requests stop
2. in-flight operation is cancelled safely
3. local durable metadata remains intact
4. client enters `stopped`

## Server State Machine

### Client Registration State

The server maintains:

- client identity
- assigned scope set
- scope set version
- schema compatibility result
- per-scope incremental progress

### Scope State

For each scope instance, the server maintains enough state to support:

- deterministic membership
- incremental change serving
- rebuild
- checksum production

### Mutation Application State

For each mutation batch, the server:

1. validates schema and protocol compatibility
2. authenticates and authorizes the request
3. applies row-level synced CRUD
4. computes resulting scope effects
5. returns canonical outcomes

### Incremental Pull State

For incremental pull, the server:

1. validates client and schema state
2. evaluates scope assignment delta if needed
3. serves incremental changes for provided scope cursors
4. advances authoritative cursor state as needed
5. returns authoritative final checksums on terminal pull pages
6. requests explicit rebuild when incremental state is invalid

### Rebuild State

For rebuild, the server:

1. validates the scope request
2. pages the authoritative contents of the scope
3. returns the final authoritative cursor and checksum

## State Transition Summary

### Client

- `uninitialized` -> `local_ready`
- `local_ready` -> `connecting`
- `connecting` -> `schema_applying`
- `connecting` -> `ready`
- `ready` -> `pushing`
- `ready` -> `pulling`
- `ready` -> `rebuilding`
- `pushing` -> `pulling`
- `pulling` -> `rebuilding`
- any active state -> `backoff`
- any active state -> `error`
- any active state -> `stopped`
- `backoff` -> `connecting` or prior active state
- `rebuilding` -> `ready`
