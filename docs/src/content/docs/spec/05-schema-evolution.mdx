# Schema Evolution

## Purpose

This document defines how Synchro handles schema discovery, schema compatibility, local migration, and rebuild during structural change.

The goal is:

- low startup cost
- low protocol chatter
- safe rollouts across multiple client versions
- automatic local migration for synced tables where possible
- explicit rebuild when structure or semantics require it

## Core Position

Synchro should treat schema evolution as a first-class product feature, not an afterthought.

The system should be better than alternatives by making schema change:

- predictable
- explicit
- cheap in the normal case
- non-destructive when older clients still exist

## Versioning Layers

Synchro uses separate version surfaces for separate concerns.

### Protocol Version

The protocol version defines wire compatibility.

It changes only when the HTTP and JSON contract changes incompatibly.

### Schema Version And Hash

The schema version and hash define the client-visible synced data shape and related local materialization contract.

They change when the client must understand a different synced schema shape or local materialization rule.

### Scope Set Version

`scope_set_version` tracks authoritative scope assignment changes for one client.

It does not represent structural schema change.

### Rebuild State

Rebuild state is per scope.

It represents data re-materialization requirements, not protocol or structural schema versioning.

## Canonical Schema Model

The server should expose a canonical portable schema manifest for synced tables.

The manifest should not be raw PostgreSQL DDL.

It should be a transport-friendly structural model that future adapters and clients can implement consistently.

At minimum, the schema manifest should define:

- synced table names
- primary keys
- updated-at semantics
- deleted-at or soft-delete semantics where applicable
- columns
- portable column types
- nullability
- local composition class, for example single-scope or multi-scope
- recommended local indexes where needed for expected query performance

Example shape:

```json
{
  "version": 8,
  "hash": "8b21d2a1",
  "tables": [
    {
      "name": "workouts",
      "primary_key": ["id"],
      "updated_at_column": "updated_at",
      "deleted_at_column": "deleted_at",
      "composition": "single_scope",
      "columns": [
        { "name": "id", "type": "uuid", "nullable": false },
        { "name": "user_id", "type": "uuid", "nullable": false },
        { "name": "name", "type": "text", "nullable": false },
        { "name": "updated_at", "type": "timestamp", "nullable": false },
        { "name": "deleted_at", "type": "timestamp", "nullable": true },
        { "name": "started_at", "type": "timestamp", "nullable": true }
      ],
      "indexes": [
        { "name": "idx_workouts_started_at", "columns": ["started_at"] }
      ]
    }
  ]
}
```

## What Counts As Schema Change

Schema version and hash should change only for client-observable structural change.

Examples that should change schema version and hash:

- add or remove synced table
- add or remove synced column
- change column type or nullability in a client-visible way
- change local composition class for a table
- change local index recommendations that the client is expected to apply

Examples that should not require schema version change by themselves:

- server-only write validation logic
- server-only auth policy changes
- scope assignment changes for one client
- row membership changes that require rebuild but do not change local table shape

Those should use scope rebuild or normal incremental sync, not fake schema version bumps.

## Schema Change Classes

Schema evolution should be classified into a small number of operational classes.

### Class 1: Server-Only Change

Examples:

- write policy adjustment
- auth policy adjustment
- server-side conflict rule refinement

Expected action:

- no schema action
- no local DDL
- optional rebuild only if materialized membership semantics changed

### Class 2: Compatible Additive Change

Examples:

- add nullable column
- add optional synced table
- add local index

Expected action:

- `replace` or `fetch`
- local DDL update
- rebuild only affected scopes if existing rows must be materialized under the new shape

### Class 3: Compatible But Rebuild-Requiring Change

Examples:

- change table from single-scope to multi-scope composition
- change membership semantics for a scope-backed table
- add an existing synced table with historical rows that must appear locally

Expected action:

- `rebuild_local`
- apply schema update
- rebuild affected scopes explicitly

### Class 4: Incompatible Structural Change

Examples:

- drop column that older clients may still read or write
- rename column in place
- change type incompatibly in place
- remove table still used by live clients

Expected action:

- use phased rollout instead of in-place breakage
- if compatibility cannot be preserved, return `unsupported`

## Connect-Time Schema Decision

Schema compatibility is decided during `POST /sync/connect`.

The server is authoritative and returns one of:

- `none`
- `fetch`
- `replace`
- `rebuild_local`
- `unsupported`

### `none`

Use when the client schema is already acceptable.

### `fetch`

Use when the client needs the canonical schema but the server wants to avoid returning a large schema body in the `connect` response.

This is a bandwidth optimization only.

It must not change semantics.

### `replace`

Use when the client can apply the new schema locally and then continue normal sync.

### `rebuild_local`

Use when the client must apply schema changes and then rebuild affected local synced state before normal incremental pull can safely resume.

### `unsupported`

Use when the client app or client runtime is too old to safely operate against the new server-side contract.

## Client Migration Rules

### Synchro Owns Synced-Table DDL

For synced tables, Synchro should own local SQLite structural migration from the canonical schema manifest.

This reduces drift between server and client.

Applications may still own:

- local-only tables
- local-only indexes
- local-only views
- app-specific caches unrelated to synced-table correctness

### Migration Ordering

When schema action is not `none`, the client should:

1. pause normal push and pull
2. persist migration intent durably
3. apply schema DDL locally
4. update schema version and hash metadata
5. rebuild affected scopes if required
6. resume normal sync

### Migration Atomicity

Where local SQLite behavior allows it, schema migration and schema metadata update should be transactional.

If fully transactional migration is not possible, the client must persist enough durable progress state to resume safely after crash.

## Queue Compatibility Rules

Schema evolution must not silently invalidate queued local mutations.

### Required Rule

If a queued mutation targets a table or column being removed or changed incompatibly, the server should not force the client through that breakage invisibly.

The rollout must instead be one of:

- phased compatible rollout
- authoritative rejection with explicit inspectable reason
- `unsupported` response requiring app upgrade

### Best Practice

Destructive schema changes should use expand-then-contract rollout:

1. add new shape compatibly
2. dual-read or dual-write as needed
3. backfill
4. migrate active clients
5. remove old shape only after old clients are retired

## Scope And Schema Interaction

Scope semantics and schema semantics are related, but they are not the same thing.

### Scope Change Without Schema Change

Examples:

- move a row from one scope instance to another
- change who is assigned a scope
- change public versus private membership logic without changing local table shape

Expected handling:

- normal incremental sync if possible
- explicit scope rebuild when needed
- no structural schema version bump just to signal data movement

### Schema Change Affecting Scope Composition

If a table changes from single-scope to multi-scope composition, or vice versa, that is a structural local materialization change.

That should bump schema version and usually require `rebuild_local`.

## Seeded And Preinitialized Databases

Seeded databases must carry:

- schema version
- schema hash
- local metadata version if distinct

At first `connect`, the server compares those values exactly as it would for any other client.

Seeded databases are warm caches only.

They do not relax schema compatibility rules.

Current release surface:

- `api/go/cmd/synchro-seed` generates a client-compatible SQLite seed from the canonical `synchro_schema_manifest()` output

## Portable Type System

The schema manifest should use a portable client-facing type system rather than backend-specific DDL strings wherever possible.

The exact type catalog can be finalized later, but it should cover at least:

- uuid
- text
- int32
- int64
- float64
- bool
- json
- blob
- date
- timestamp
- decimal

If a server-side type cannot be represented safely in the portable schema model, the server must either:

- map it explicitly to a supported portable type, or
- reject it as unsupported for synced-table materialization

## Rollout Patterns

### Preferred Pattern: Expand, Migrate, Contract

This should be the default guidance for all non-trivial schema evolution.

Example:

1. add `session_started_at`
2. keep `started_at`
3. dual-write or backfill
4. update clients to read new field
5. retire old field only after old clients are gone

### Avoid In-Place Rename

Column rename should not be treated as a magical first-class sync diff.

In practice, a rename across real deployed clients behaves like add-new then retire-old.

### Avoid In-Place Incompatible Type Change

Type changes that alter client serialization or queued mutation compatibility should use phased rollout, not forced in-place mutation of old clients.

## Performance Rules

Schema evolution must preserve the performance goals from `00-principles`.

That means:

- schema checks should be cheap hash and version comparisons at `connect`
- full schema bodies should be sent only when needed
- local DDL should stay off the steady-state hot path
- rebuild should target affected scopes only
- schema change should not imply full local wipe unless correctness truly requires it

## Worked Examples

### Example 1: Add Nullable Column

Change:

- add nullable `notes` column to `workouts`

Expected handling:

- bump schema version and hash
- `replace` or `fetch`
- local SQLite adds column
- no rebuild unless existing authoritative rows need special materialization treatment

### Example 2: Add Public Table With Existing Data

Change:

- add synced `exercise_categories` table with existing server rows

Expected handling:

- bump schema version and hash
- apply schema locally
- rebuild affected public scopes so historical rows appear

### Example 3: Public And Private Union Refactor

Change:

- split `exercises` from one coarse scope design into `exercises_public` and `exercises_user:<id>`

Expected handling:

- no fake global wipe
- if local composition metadata changes, bump schema version
- rebuild affected scopes
- preserve local rows that still have valid provenance

### Example 4: Breaking Rename

Change:

- rename `started_at` to `session_started_at`

Expected handling:

- do not rename in place for live clients
- use phased rollout
- return `unsupported` only if compatibility window is intentionally closed
