Synchro’s release architecture is intentionally narrow:

- `extensions/synchro-core` owns shared deterministic sync semantics
- `extensions/synchro-pg` owns PostgreSQL-near execution, scope materialization, and synced CRUD application
- `api/go` owns transport, auth integration, and HTTP translation
- native clients own local SQLite, queueing, scheduling, and safe apply
- React Native is a bridge over the native clients, not an independent sync engine

## Why this shape

This architecture exists to keep:

- sync semantics close to the data
- the HTTP layer thin
- native SDK behavior consistent
- the hot path low in round trips

## Core design choices

- deterministic server-defined scopes
- extension-side change capture and materialization
- per-scope cursors and explicit rebuild
- full synced CRUD for synced tables
- SQL migration based configuration, with structured defaults and custom SQL escape hatches

## Scope model at a glance

The most important selective-sync rule is still:

- optimize for low row-change fanout, not low total scope count

Recommended baseline:

- private data lives in private scopes such as `documents_user:{user_id}`
- shared data lives in shared scopes such as `templates_public`
- one local SQLite table may be fed by both shared and private scopes

This page is only the mental model. Detailed operational guidance, grouping rules, and anti-patterns live in [Scope modeling](/synchro/architecture/scope-modeling/).

## Portable seeds at a glance

Portable seeds let you bundle:

- the canonical synced-table schema
- optional server-declared portable scopes
- continuation metadata so clients can sync forward after login

They are generated by trusted tooling and do not accept JWT input.

Detailed rules live in [Portable seeds](/synchro/architecture/portable-seeds/).

## Auth integration at a glance

The Go adapter supports:

- trusted upstream auth
- direct JWT validation

For most product APIs, trusted upstream auth is the better default because product-specific identity logic stays in the host API instead of the sync adapter.

Detailed wiring guidance lives in [Auth integration](/synchro/architecture/auth-integration/).
